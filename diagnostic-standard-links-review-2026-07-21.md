# Ревью связей диагностик с пунктами стандартов

Дата проверки: 2026-07-21.

## Вывод

Связи нельзя считать полностью корректными.

На уровне **страницы стандарта** текущее покрытие в основном достоверно: все
706 локальных связей технически целы, а для АПК все 439 связей из исходного
индекса присутствуют. Но обещанный переход к **пункту стандарта** фактически не
реализован: ни одна из 706 ссылок с диагностик не содержит якоря пункта. При
импорте точный пункт из upstream теряется, а обратные ссылки часто складываются
в общий блок `Проверки` в конце страницы.

Кроме этого обнаружены пропущенные связи с целыми страницами стандартов:

- 10 подтверждённых связей v8-code-style;
- 28 предметных связей BSL Language Server;
- ещё 44 ссылки BSL Language Server на `std456`, которые есть в upstream, но
  похожи на ошибочный массовый boilerplate и не должны импортироваться без
  исправления первоисточника.

Цикл проверки для каждой связи:

1. разобрать ссылку со страницы диагностики;
2. найти обратную ссылку на странице стандарта;
3. проверить существование обеих страниц и симметрию связи;
4. сравнить номер стандарта с официальным upstream;
5. если upstream сохранил номер пункта, сравнить его с положением блока
   `Проверки` в текущем стандарте;
6. спорные случаи перечитать по тексту диагностики и самого стандарта.

Upstream-срезы:

- v8-code-style и индекс АПК: `c8fe7932babf718c0ace3cf836a99d6a3b98d098`;
- BSL Language Server: `f4616cda8a216789ee40529ed857e614b9e2ea25`.

## Структурная проверка

- прямых связей диагностика → стандарт: 706;
- обратных связей стандарт → диагностика: 706;
- несимметричных связей: 0;
- ссылок на отсутствующие страницы: 0;
- ссылок на конкретный пункт (`#11`, `#243` и т. п.): 0.

Последний результат принципиален: симметрия доказывает только целостность
графа, но не правильность смысла. Текущие тесты также проверяют каталог,
канонические идентификаторы и поиск, но не семантику связи и не номер пункта.

### АПК

Номера страниц из официального индекса совпадают. Точное место не хранится ни
в upstream, ни в локальной ссылке. На страницах с несколькими независимыми
правилами это уже даёт неверную навигацию. Например:

- все 18 проверок АПК для `std474` собраны после пункта 4, хотя относятся к
  пунктам 1.1–3.2;
- все 17 проверок АПК для `std488` собраны после пункта 6.5, хотя относятся к
  разделам 2–5;
- `acc:89` (использован `Перейти`) и `acc:249` (ограничение веб-клиента)
  должны вести соответственно к `std547/1` и `std547/2`, а не в один общий
  блок;
- `acc:293` относится к `std640/6.2` и `std693/1`, но обе связи сейчас только
  страничные.

## Корневая причина

Связь хранится как пара `(диагностика, стандарт)`. Для задачи, заявленной в
`docs/mcp.md`, этого недостаточно: нужна как минимум тройка
`(диагностика, стандарт, пункт)` и происхождение связи.

Из-за потери пункта возникают сразу четыре эффекта:

1. ссылка с диагностики открывает начало длинной страницы;
2. генератор не знает, возле какого правила размещать обратную ссылку;
3. тесты могут доказать симметрию, но не смысл;
4. при изменении нумерации нельзя отличить устаревшую связь от корректной.

## План исправления причины

1. Ввести единый реестр связей с полями `family`, `diagnostic_id`,
   `standard_id`, `clause`, `source_url`, `source_revision`, `confidence`.
2. Нормализовать все варианты upstream URL, включая `/content/NNN/hdoc#X`, и
   выдавать ошибку импорта для неизвестной формы вместо молчаливого пропуска.
3. Генерировать ссылку с диагностики на реальный HTML-якорь пункта: например,
   `../../std/455.md#243` для пункта 2.4.3.
4. Генерировать обратный блок возле соответствующего пункта из того же реестра,
   а не поддерживать две стороны вручную.
5. Добавить тесты на полноту upstream, допустимость пункта, существование якоря,
   семантические исключения и отсутствие generic-связей без подтверждения.
6. После миграции исправить перечисленные пропуски и спорные связи отдельными
   пакетами по семействам; `std456` не импортировать до исправления upstream.

Такой порядок устраняет источник расхождений. Простое ручное перемещение
текущих блоков оставит прежнюю модель данных и ошибки вернутся при следующей
синхронизации.
